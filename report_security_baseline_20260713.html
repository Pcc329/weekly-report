Section 01
背景與範圍

本報告回應近期會議提出之平台資訊安全議題，針對「產業策略智庫」線上系統（solution-finder-gray.vercel.app）進行外部安全檢測，建立基線紀錄，並完成第一波可立即實施的補強項目。

檢測範圍涵蓋三個層面：網路基礎設施層（由院內其他單位以 Tenable Nessus 協助掃描）、TLS 傳輸層（Qualys SSL Labs 憑證驗證），以及應用回應層（Mozilla HTTP Observatory、SecurityHeaders、Google Lighthouse）。所有檢測均為被動式掃描或平台方協助執行，未對系統發送侵入性攻擊請求。

一頁結論：網路層與 TLS 層檢測結果乾淨（零弱點、憑證鏈驗證通過）；應用回應層原缺五個 HTTP 安全標頭，已於當日補齊四個並完成部署驗證，評分由 D 提升至 A（SecurityHeaders）、C- 提升至 B（Observatory）。惟外部掃描無法涵蓋應用架構層風險（API 認證、流量限制、個資防護），該部分依既定計畫由「安全層一」與資料庫遷移（Row Level Security）處理，於使用者範圍擴大前完成。

07-13 下午

07-13 下午
正式站複掃：SecurityHeaders 評分 A、Observatory 評分 B（70/100，8/10 項通過）、Lighthouse Best Practices 77 分。

安全標頭補強前補強後作用
Strict-Transport-Security已具備維持強制 HTTPS 連線（Vercel 平台預設，max-age 兩年 + preload）

Permissions-Policy缺失已補停用相機、麥克風、定位等本平台未使用之瀏覽器權限
Content-Security-Policy缺失規劃中限制可載入之外部資源來源；因涉及全站 CDN 白名單盤點，另案以 Report-Only 模式先行試運行後再正式啟用

本次補強僅修改部署設定檔（vercel.json），未變動任何 API 邏輯與前端程式碼，變更風險受控。CSP 為剩餘唯一缺失標頭，因設定錯誤將直接造成頁面外部資源（前端框架、圖表函式庫等 CDN）載入失敗，採「先觀察後啟用」的兩階段方式另案處理。

Section 04
網路層掃描結果解讀（Tenable Nessus）

本項掃描由院內其他單位主動協助執行（Tenable Nessus 10.11.2，Advanced Scan 原則、Safe Checks 啟用、未含網頁應用測試），掃描時間 2026-07-13 11:26–11:44，歷時約 17 分鐘。結果為 0 Critical / 0 High / 0 Medium / 0 Low，21 筆偵測項目均為資訊性（INFO）等級。

掃描發現內容解讀
開放埠 僅 80（HTTP 轉址）與 443（TLS 1.3） 正常 攻擊面最小化，無其他服務暴露
HSTS 標頭 max-age 兩年 + includeSubDomains + preload 正常 與應用層掃描結果一致
掃描防護觸發 443 埠回應 403，回應標頭含 X-Vercel-Mitigated: challenge 正面訊號 託管平台之機器人防護偵測到自動化掃描並啟動驗證挑戰，屬平台內建防護能力的實地驗證
作業系統指紋「SonicWALL」 OS 辨識信心度僅 56% 誤判 為 Vercel 邊緣網路設備之 TCP 指紋特徵，非本系統實際運行環境，不需處理
其他主機名清單（mta / pop / webmail 等 *.vercel.app） 29 個指向相同節點的主機名 與本系統無關 為 vercel.app 共用網域下的其他主機名，非本平台之攻擊面
STS 轉址狀態碼 port 80 轉址使用 308 而非標準草案之 301 無風險 308 為語意更嚴格的永久轉址，功能等價

綜合解讀：本系統採無伺服器（Serverless）架構託管，不存在自行維運的主機與服務埠，網路層攻擊面本質上由託管平台管理。本次掃描驗證了此架構特性——無可利用之網路層弱點，且平台自動化防護實際發揮作用。惟同一特性也意味網路層掃描無法觸及應用邏輯與資料層，該範圍的風險評估見次章。

Section 05
檢測範圍的界限：外部掃描測不到的風險

本次三層掃描的共同性質為「由外部觀察系統回應」。掃描結果良好代表網路層與回應層沒有低級失誤，但不代表系統整體安全——應用架構層的既知風險不在任何外部掃描工具的偵測範圍內，需明確區分，避免以掃描評分推論整體安全狀態。

🔑
API 無身分驗證
各 API 端點目前無存取控管，知悉網址即可呼叫。於使用者範圍擴大前需建立認證機制。
🚦
無流量限制（Rate Limiting）
API 層無請求頻率限制，且底層 Airtable 免費方案有每秒 5 請求之硬上限，多人同時使用時存在服務中斷風險。
📇
聯絡人個資（個資法）
Contacts 資料表含 677 筆廠商真實聯絡電話，屬個資法規範範圍，需比照資料分級原則以限閱等級控管。
🛡️
欄位白名單為唯一防線
敏感欄位（如案例企業真實名稱）目前僅依賴 API 回傳欄位白名單過濾，缺乏資料庫層級的存取控制作為第二道防線。
解讀原則：掃描評分 A 反映的是「HTTP 回應層設定完善」，上述四項架構層風險均非外部掃描可偵測，其處理依既定計畫進行（見 Section 06），並作為使用者範圍由試行小組擴大至完整群組前的前置條件。

治理面向補充：本次網路層掃描由院內其他單位協助執行，掃描時間、範圍與執行單位均已留存紀錄。後續如再執行主動式掃描，將先確認託管平台服務條款容許範圍並比照留痕。

Section 07
結語

本次檢測建立了平台的三層資安基線：網路層零弱點、TLS 憑證驗證通過、應用回應層於當日完成補強並複驗（D→A、C-→B）。外部可見面向的安全狀態已有量化紀錄可供追蹤。

同時，檢測過程也明確了外部掃描的界限——真正的風險重心在應用架構層（API 認證、流量限制、個資防護），該部分與平台既定的分階段開放規劃綁定：先以小規模試行驗證使用情境，於安全層一與資料庫遷移完成後，再開放予完整使用者群組。掃描評分是外層健康度的指標，架構層工程才是資料資產保護的根本。
