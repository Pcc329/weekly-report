Section 01
背景與範圍
  • 目的：回應近期會議提出之平台資訊安全議題，建立檢測基線並完成第一波補強
  • 對象系統：產業策略智庫線上平台（solution-finder-gray.vercel.app）

安全標頭補強前補強後作用
Strict-Transport-Security已具備維持強制 HTTPS 連線（Vercel 平台預設，max-age 兩年 + preload）
X-Content-Type-Options缺失已補阻止瀏覽器猜測內容類型，防範 MIME 混淆攻擊
X-Frame-Options缺失已補禁止頁面被嵌入 iframe，防範點擊劫持
Referrer-Policy缺失已補限制跨站連結時傳遞的來源資訊
Permissions-Policy缺失已補停用相機、麥克風、定位等本平台未使用之瀏覽器權限
Content-Security-Policy缺失規劃中限制可載入之外部資源來源；因涉及全站 CDN 白名單盤點，另案以 Report-Only 模式先行試運行後再正式啟用
  • 變更範圍：僅修改部署設定檔（vercel.json），未變動任何 API 邏輯與前端程式碼，風險受控
  • 剩餘缺項：CSP 為唯一未補標頭——設定錯誤會直接造成 CDN 資源載入失敗（頁面白屏）
  • CSP 處理方式：另案採兩階段——先以 Report-Only 模式觀察一週，白名單確認無遺漏後轉正式啟用
Section 04
網路層掃描結果解讀（Tenable Nessus）
  • 執行單位：院內其他單位主動協助執行
  • 工具與設定：Tenable Nessus 10.11.2 · Advanced Scan · Safe Checks 啟用 · 未含網頁應用測試
  • 掃描時間：2026-07-13 11:26–11:44（約 17 分鐘）
  • 結果：0 Critical / 0 High / 0 Medium / 0 Low，21 筆均為資訊性（INFO）項目
掃描發現內容解讀
開放埠 僅 80（HTTP 轉址）與 443（TLS 1.3） 正常 攻擊面最小化，無其他服務暴露
HSTS 標頭 max-age 兩年 + includeSubDomains + preload 正常 與應用層掃描結果一致
掃描防護觸發 443 埠回應 403，回應標頭含 X-Vercel-Mitigated: challenge 正面訊號 託管平台之機器人防護偵測到自動化掃描並啟動驗證挑戰，屬平台內建防護能力的實地驗證
作業系統指紋「SonicWALL」 OS 辨識信心度僅 56% 誤判 為 Vercel 邊緣網路設備之 TCP 指紋特徵，非本系統實際運行環境，不需處理
其他主機名清單（mta / pop / webmail 等 *.vercel.app） 29 個指向相同節點的主機名 與本系統無關 為 vercel.app 共用網域下的其他主機名，非本平台之攻擊面
STS 轉址狀態碼 port 80 轉址使用 308 而非標準草案之 301 無風險 308 為語意更嚴格的永久轉址，功能等價
  • 架構特性:系統採無伺服器（Serverless）託管,無自行維運的主機與服務埠,網路層攻擊面由託管平台管理
  • 本次驗證:無可利用之網路層弱點,且平台自動化防護實際發揮作用（掃描遭偵測攔截）
  • 界限:同一架構特性意味網路層掃描無法觸及應用邏輯與資料層,該範圍見次章
Section 05
檢測範圍的界限：外部掃描測不到的風險
  • 三層掃描的共同性質：由外部觀察系統回應，僅能驗證「網路層與回應層無低級失誤」
  • 關鍵區分：掃描結果良好 ≠ 系統整體安全——應用架構層的既知風險不在任何外部掃描工具的偵測範圍內
  • 解讀提醒：不應以掃描評分推論整體安全狀態
🔑
API 無身分驗證
各 API 端點目前無存取控管，知悉網址即可呼叫。於使用者範圍擴大前需建立認證機制。
🚦
無流量限制（Rate Limiting）
API 層無請求頻率限制，且底層 Airtable 免費方案有每秒 5 請求之硬上限，多人同時使用時存在服務中斷風險。
📇
聯絡人個資（個資法）
Contacts 資料表含 677 筆廠商真實聯絡電話，屬個資法規範範圍，需比照資料分級原則以限閱等級控管。
🛡️
欄位白名單為唯一防線
敏感欄位（如案例企業真實名稱）目前僅依賴 API 回傳欄位白名單過濾，缺乏資料庫層級的存取控制作為第二道防線。
解讀原則：掃描評分 A 反映的是「HTTP 回應層設定完善」，上述四項架構層風險均非外部掃描可偵測，其處理依既定計畫進行（見 Section 06），並作為使用者範圍由試行小組擴大至完整群組前的前置條件。
Section 06
後續處理計畫
優先序項目內容時點
已完成 安全標頭第一波 四個零風險標頭部署上線，掃描複驗通過 2026-07-13
近期 CSP 第二波 盤點全站外部 CDN 資源來源，以 Content-Security-Policy-Report-Only 模式試運行一週，確認白名單無遺漏後轉正式啟用 標頭第一波穩定後
擴大前必要 安全層一 API 身分驗證、流量限制、敏感欄位存取驗證——使用者範圍擴大前之前置條件 試行階段期間規劃
擴大前必要 資料庫遷移（Supabase / PostgreSQL） 以 Row Level Security 建立資料庫層級存取控制（補上第二道防線），同時解決 Airtable 請求頻率上限問題 與安全層一協同規劃
暫緩 主動式滲透測試（OWASP ZAP 等） 涉及兩項授權議題：Vercel 服務條款對侵入式掃描之限制、Airtable 為第三方系統非自有資產。待安全層一完成後再評估必要性與授權取得方式 安全層一之後
  • 掃描留痕：本次網路層掃描之時間、範圍與執行單位均已留存紀錄
  • 後續原則：如再執行主動式掃描，先確認託管平台服務條款容許範圍，並比照留痕
Section 07
結語
  • 已完成：三層資安基線建立——網路層零弱點、TLS 憑證驗證通過、應用回應層當日補強並複驗（D→A、C-→B），外部可見面向已有量化紀錄可追蹤
  • 已明確：外部掃描的界限——風險重心在應用架構層（API 認證、流量限制、個資防護）
  • 處理路徑：與分階段開放規劃綁定——小規模試行驗證使用情境 → 安全層一與資料庫遷移完成 → 開放完整使用者群組
  • 核心觀念：掃描評分是外層健康度指標，架構層工程才是資料資產保護的根本
