SECTION 01

為什麼要改造週報

先解決資訊管理問題，再選擇工具。週報真正的成本，往往不是撰寫文字，而是每週重排、重找、重複確認與無法延續。

傳統週報常見情況

  • 每週重新排版，時間花在格式而非內容
  • Word、郵件與簡報散落，歷史資料難追蹤
  • 主管只能看到當週，缺乏長期脈絡
  • 數字靠人工更新，容易前後不一致
  • 工作項目很多，卻看不出成果、風險與下一步

改造後希望達成

  • 單一入口：以永久網址查看最新進度
  • 固定骨架：每週只更新內容，不重做版面
  • 狀態透明：成果、問題、數據、決策需求並列
  • 歷史可查：每期封存，形成專案時間軸
  • 持續累積：週報成為專案知識庫的一部分

一句話定位

讓週報從「交作業」轉型為「專案單一資訊入口（Single Source of Truth）」。

SECTION 03

週報系統的四層架構

把內容、呈現、數據與歷史拆開管理，才能降低每週維護成本，也避免所有資訊綁死在單一文件裡。

LAYER 01內容層 Markdown

  • 以 report.md 集中維護當週內容
  • 章節固定：摘要、完成事項、數據、風險、下週規劃
  • 不需在 HTML 中逐段修改文字

LAYER 02展示層 一次建置

  • 由固定網頁框架呈現週報內容
  • 支援桌機與手機閱讀
  • 統一字級、表格、標籤與視覺層次

LAYER 03數據層 可選配

  • KPI 可由系統 API 即時取得
  • 降低人工複製數字與版本落差
  • 僅將適合公開或內部呈現的數據放入頁面

LAYER 04歷史層 可追蹤

  • 過去週報依日期封存
  • 首頁集中顯示歷史紀錄
  • 可回查專案變化、決策與工作累積

維護邏輯

一次建立展示框架，每週只更新結構化內容；能自動取得的數據由系統提供，舊週報則持續封存。

AI 適合負責 助理角色

  • 把零散筆記整理成固定章節
  • 濃縮冗長敘述，突出成果與影響
  • 協助統一用詞、語氣與格式
  • 檢查是否缺少風險、下一步或待決策事項

專案負責人必須確認 不可外包

  • 任務是否真的完成，以及完成定義
  • 數據、日期、成本與引用來源是否正確
  • 問題優先度及對專案的實際影響
  • 哪些內容適合對內分享或公開展示
SECTION 05

可複製的週報內容骨架

其他同仁不必先學 GitHub。第一步是先採用一致的資訊骨架，讓主管能快速回答：本週完成了什麼、數字如何、卡在哪裡、下週要做什麼。

01 本週摘要

以 3–5 句說明最重要的成果、變化與風險。

02 本週完成事項

以任務、狀態及具體成果呈現，不只寫「已處理」。

03 關鍵數據

列出本週、上週與差異，讓趨勢一眼可見。

04 問題與風險

標示影響、處理方式與優先度，避免只描述現象。

05 待協助／待決策

明確寫出需要誰協助、需要決定什麼及期待時點。

06 下週規劃

聚焦 3–5 項可驗證的下一步，與本週風險相互對應。

SECTION 06

建議的 15–20 分鐘分享流程

段落時間分享重點現場展示
1. 為什麼改造3 分鐘傳統週報的重工、資訊分散與追蹤困難從主管閱讀需求切入
2. 設計原則3 分鐘固定結構、內容與版面分離、數據自動化、歷史追蹤四層架構圖
3. 每週操作5 分鐘從平日記錄到發布與封存，以及 AI 的角色report.md 與網頁對照
4. 成果展示3 分鐘KPI、完成事項、風險、歷史週報及手機版實際 weekly.html
5. 團隊採用3–5 分鐘依工具熟悉度分級導入，不要求所有人一次到位Level 1–3 路徑

分享時的主軸

不要把分享變成 HTML 語法教學。先說明資訊怎麼被結構化、怎麼降低每週維護成本、怎麼讓主管更快掌握專案，技術只是支撐方法的工具。

SECTION 07

團隊導入可分三個層級

組織推廣不應假設所有人都熟悉程式。先建立共同模板，再逐步增加網頁呈現與自動化能力。

LEVEL 01 · 基礎
模板版
適合一般同仁
使用 Word、Google 文件或 Markdown 固定模板
先建立共同資訊語言
→
LEVEL 02 · 進階
網頁版
適合熟悉數位工具者
固定網頁讀取週報內容
建立永久入口與歷史查詢
→
LEVEL 03 · 自動化
系統版
適合已有資料來源的專案
串接 API、KPI 與自動封存
降低重複輸入與版本落差

建議導入順序

先統一「寫什麼」，再決定「用什麼工具寫」；先驗證閱讀價值，再投入自動化。

SECTION 08

品質治理與注意事項

事實責任

AI 可以協助寫得清楚，但完成狀態、數字與風險判斷仍由專案負責人背書。

內容分級

公開頁面、內部頁面與敏感資訊應分開管理，避免把聯絡資料、內部成本或未定案內容誤公開。

文件同步

架構變更時，README、操作說明與範例也要同步更新，否則工具可用但知識無法交接。

建議下一步：先以一個小組進行模板試行

不急著要求每位同仁建立網站，可先用相同模板完成 2–3 週週報，再觀察主管閱讀效率與同仁維護負擔。

  • 第一步：確認組內共用的週報章節與欄位
  • 第二步：選擇 1–2 個專案試用固定模板
  • 第三步：收集主管最常追問的資訊，調整模板
  • 第四步：有明確需求後，再評估網頁化與 API 自動化
